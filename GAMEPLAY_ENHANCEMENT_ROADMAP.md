# 《危城无双之三国群英》重制版玩法增强总纲

> 文档状态：规划中
> 版本：1.0
> 更新日期：2026-07-22
> 适用范围：game/ H5 工程
> 预计总预算：完整方案 141 万～242 万 Token，建议预留 190 万 Token

## 1. 文档目的

本文是玩法增强工程的总控文件，用来固定产品边界、跨阶段技术约定、阶段依赖、验收门槛和 AI 协作方式。具体功能不在本文展开，分别由三个阶段方案维护：

- [第一阶段：战术基础与局内决策](docs/gameplay/PHASE_1_TACTICAL_FOUNDATION.md)
- [第二阶段：武将、名将与阵营特色](docs/gameplay/PHASE_2_HEROES_AND_BOSSES.md)
- [第三阶段：历史事件与重玩模式](docs/gameplay/PHASE_3_REPLAY_MODES.md)

本文不是一次性需求清单。每完成一个阶段，都要根据实玩结果更新后续阶段的边界和预算，避免过早锁死设计。

## 2. 总体目标

在不破坏原版还原成果的前提下，为重制版增加可重复游玩的强化规则，使玩家持续面对以下决策：

- 立即出兵争取奖励，还是等待并完善防线；
- 选择稳定的通用强化，还是承担代价追求强力组合；
- 围绕武将、机关联动和敌方名将调整布局；
- 在历史事件、挑战条件和无尽压力下改变常规解法。

最终应形成两种并存体验：

1. **经典规则**：继续承担原版还原职责，默认保持现有数值、流程和战斗行为。
2. **强化规则**：在经典战斗之上启用军议、联动、将技、名将特性、事件和挑战模式。

## 3. 明确不做的事情

- 不把武将改成需要自由寻路和实时操控的地图单位。
- 不用大量同质化新塔替代机关组合设计。
- 不让强化规则的代码无条件进入经典模式结算。
- 不以永久攻击力叠加作为主要局外成长。
- 不在第三阶段之前建设联网排行榜或账号系统。
- 不承诺新增商业美术、配音和服务器；需要时另立资源方案。
- 不因玩法增强顺手重写所有原版流程或反编译数据。

## 4. 当前工程基线

截至 2026-07-22，game/src 约有 13,017 行 TypeScript，game/tests 约有 1,428 行测试。现有能力包括：

- 9 张战役地图、原版波次和名将波；
- 11 类建筑、升级、道路朝向、科技建筑和武将觉醒；
- 火焰、寒冰、麻痹、中毒、减速等独立状态；
- 滚油遇火、沸水灭火、火冰互斥等原版联动；
- 1～3 倍速、手动出兵、及时出兵判定；
- 首次通关、无人入城、及时出兵记录；
- 军令、正攻/奇谋胜利、失败救援和沙盒解锁；
- 单战斗存档、独立军令档案和 localStorage 持久化；
- 固定 10Hz 战斗逻辑和较完整的塔、状态、Boss 回归测试。

本项目的主要技术风险不是缺少基础系统，而是核心协调逻辑集中在 Game.ts、Tower.ts 和 Enemy.ts。新增玩法必须优先拆出独立模块，避免继续扩大三个大文件。

## 5. 统一产品约定

### 5.1 规则集与游戏模式分离

现有 gameModeT 继续表达王者之路/自由模式，不用于表达经典/强化规则。新增独立概念：

~~~ts
export type RulesetId = 'classic' | 'enhanced';
~~~

所有新增功能必须明确声明适用规则集。经典规则下，新系统应返回空效果或完全不创建。

### 5.2 成绩分类

成绩至少按以下维度区分：

- 规则集；
- 阵营；
- 战役；
- 是否使用军令；
- 挑战或无尽种子。

使用军令可以正常推进和保存，但正式勋章、每日挑战成绩和无尽纯净记录只能由未使用军令的对局获得。现有“正攻取胜/奇谋取胜”语义保持不变。

### 5.3 随机性

- 经典规则继续保留当前原版随机行为，除非另有还原修复任务。
- 强化规则的军议、挑战、事件和无尽波次必须使用可注入的种子随机数。
- 不允许直接散落新的 Math.random 调用。
- 同一版本、同一种子、同一输入序列必须得到相同的强化内容。
- 随机算法及内容池版本要写入运行存档，避免版本更新后无法恢复。

### 5.4 数值和内容数据化

军议、将技、名将特性、阵营政令、事件和挑战条件应存放在独立数据模块中。系统代码只解释配置，不用大量战役编号或英雄编号分支硬编码规则。

推荐新增目录：

~~~text
game/src/enhancement/
├── Ruleset.ts
├── SeededRng.ts
├── RunState.ts
├── CombatEvents.ts
├── modifiers/
├── skills/
├── bosses/
├── events/
└── challenges/

game/src/data/enhancement/
├── councilData.ts
├── reactionData.ts
├── heroSkillData.ts
├── bossTraitData.ts
├── factionDecreeData.ts
├── levelEventData.ts
└── challengeData.ts
~~~

最终目录可在第一阶段技术设计时微调，但必须维持“原版数据”和“增强数据”边界。

### 5.5 存档兼容

当前战斗存档版本为 1，加载器会拒绝其他版本。第一阶段必须先建立显式迁移机制，再增加强化字段。

最低要求：

- 现有 v1 存档可无损升级；
- 旧军令档案继续有效；
- 经典战斗存档不必包含强化运行状态；
- 强化战斗保存种子、内容池版本、军议选择、计时、勋章条件和事件状态；
- 不允许用“版本不一致直接忽略”处理未来的兼容升级；
- 每次迁移都有纯逻辑测试。

### 5.6 战斗事件与统计

第二、三阶段都会依赖可靠的战斗事件。第一阶段建立最小事件层，至少覆盖：

- 波次规划、开始、完成；
- 敌人生成、受伤、死亡、入城；
- 塔建造、升级、出售、觉醒；
- 状态施加、刷新、移除和联动；
- 军令使用、关卡结束。

事件层只传递事实，不能直接承担渲染。统计、军议、勋章、将技和事件系统分别订阅需要的事件。

## 6. 阶段路线

| 阶段 | 核心产出 | 建议 Token | 开始条件 | 完成门槛 |
|---|---|---:|---|---|
| 第一阶段 | 强化规则壳、波次情报、连战奖励、勋章、军议、增强联动、战报 | 35 万 | 当前经典版构建和测试通过 | 经典无回归，强化模式可完整通关至少 3 张图 |
| 第二阶段 | 主动将技、6 名特色名将、阵营政令 | 60 万 | 第一阶段系统和数值稳定 | 6 名武将、6 名敌将、3 阵营均有清晰差异 |
| 第三阶段 | 事件框架、9 图事件、无尽、每日挑战和计分 | 95 万 | 第二阶段战斗接口冻结 | 事件战役可通关，无尽可稳定运行 200 波，每日种子可复现 |

### 6.1 阶段依赖

~~~text
第一阶段
  ├─ 规则集隔离 ─────────────┐
  ├─ 种子随机 ───────────────┤
  ├─ 战斗事件与运行状态 ──────┼─> 第二阶段将技/名将/政令
  └─ 勋章与统计 ──────────────┘              │
                                               v
第三阶段事件、无尽、每日挑战、统一计分与记录
~~~

第二阶段不得绕过第一阶段事件和修正器接口直接修改 Game.ts 中的金币或 Tower.ts 中的伤害。第三阶段不得为每张地图复制一套独立状态机。

## 7. 里程碑执行规则

每个阶段由多个 4 万～10 万 Token 的里程碑组成。单个里程碑必须满足：

1. 有明确输入、输出和非目标。
2. 改动前记录经典模式基线测试。
3. 优先新增独立模块，再在现有系统增加最小接入点。
4. 数据、逻辑、渲染和存档分别实现并测试。
5. 完成 npm test 与 npm run build。
6. 涉及界面的功能必须完成桌面和触屏尺寸视觉验收。
7. 更新对应阶段文档中的状态、实际改动和剩余风险。
8. 未经用户明确要求，不自动提交、推送或发布。

## 8. AI 开发工作流

### 8.1 每个里程碑开始前

- 阅读本总纲和当前阶段方案；
- 检查 Git 状态，保护用户已有修改；
- 核对相关源码和现有测试，不能仅按文档猜测；
- 将里程碑从“待开发”改为“开发中”；
- 给出本轮预计 Token 上限和退出条件；
- 若实现发现产品边界需改变，先更新方案并说明原因。

### 8.2 实现期间

- 尽量把纯逻辑做成无 DOM、无 Canvas 依赖的模块；
- 先写确定性测试，再接入 UI；
- 每次只接入一个系统，避免跨多个里程碑的大爆炸修改；
- 超出里程碑范围的问题记录到“后续事项”，除非阻塞当前验收；
- 不用大规模机械重写覆盖用户的并行改动；
- 连续调试无进展时缩小到最小复现，不通过堆条件掩盖问题。

### 8.3 每个里程碑结束时

- 报告变更文件、行为变化和兼容性；
- 报告实际执行的测试和未执行的测试；
- 记录数值决策、已知问题和下一里程碑入口；
- 将验收项逐条标记，不以“编译成功”代替玩法验收；
- 若使用了 Token 目标预算，记录实际消耗和偏差原因。

## 9. 测试策略

### 9.1 自动测试层次

1. **纯逻辑测试**：种子随机、波次计划、修正器、勋章、计分、迁移。
2. **系统测试**：EnemySystem、TowerSystem、将技、Boss、事件之间的交互。
3. **存档往返测试**：保存、加载、迁移后关键运行状态一致。
4. **确定性回放测试**：相同种子和输入得到相同选择、事件和计分。
5. **长时间测试**：无尽模式至少模拟 200 波，不出现数值溢出、内存持续增长或软锁。

当前单一 timing.test.ts 已较大。第一阶段应调整测试入口，让增强测试按主题拆分文件，同时保留一个命令执行全部测试。

### 9.2 经典模式保护

每阶段必须证明：

- 经典规则没有军议、增强反应、主动将技、事件修正或挑战奖励；
- 原版塔伤害、冷却、状态免疫和波次条件测试继续通过；
- 同一旧存档升级后，金币、城防、塔、波次和战役进度一致；
- UI 新入口不会阻塞原版标题、国家、升级路线和战役流程。

### 9.3 视觉与操作验收

- 240×320 逻辑画布信息不重叠；
- 桌面鼠标、键盘和触屏均能完成关键选择；
- 1～3 倍速下提示、冷却和暂停逻辑一致；
- 名将、联动和事件都有发生前提示与发生后反馈；
- 色彩之外至少提供文字或图形差异，不能仅靠颜色表达状态。

## 10. 性能预算

- 正常战斗不因增强规则产生每帧全表扫描；
- 联动检测优先随命中事件触发，不遍历所有塔与所有敌人的笛卡尔积；
- 无尽模式仅保留当前波、必要历史统计和有限事件日志；
- 战斗事件订阅在关卡结束时释放；
- Canvas 新特效要复用缓存，避免每帧创建离屏画布或图片对象；
- 低端移动设备在 3 倍速和 80 个敌人上限附近仍应保持可操作。

## 11. 预算与变更控制

### 11.1 预算口径

Token 预算包含代码阅读、推理、修改、测试、调试和交付说明，是规划用区间，不是账单承诺。建议为测试与返工预留阶段预算的 25%～35%。

### 11.2 触发重新估算的情况

- 要求新增自由移动武将；
- 加入联网排行榜、账号或云存档；
- 要求全部 33 名己方武将拥有独立主动技能；
- 要求所有敌将都有独立动画和配音；
- 地图事件需要运行时改写寻路拓扑；
- 经典模式也要采用强化数值；
- 新增关卡编辑器或玩家创意工坊。

出现以上任一项，应暂停对应里程碑，单独立项和估算。

## 12. 总体验收标准

全部三个阶段完成时，必须满足：

- [ ] 玩家可明确选择经典规则或强化规则。
- [ ] 经典规则通过原有全部自动测试和一轮战役流程验收。
- [ ] 强化规则下，每两波左右至少出现一次有意义的战术决策。
- [ ] 至少 6 名己方武将拥有主动技能。
- [ ] 至少 6 名敌方名将拥有可预告、可应对的独立机制。
- [ ] 蜀、魏、吴的强化打法有可感知差异。
- [ ] 9 张战役地图至少各有一个专属事件或目标。
- [ ] 无尽模式可稳定运行 200 波。
- [ ] 每日挑战在相同日期和内容版本下完全可复现。
- [ ] 军令辅助成绩与纯净成绩分开记录。
- [ ] 旧存档迁移有测试，不丢失原版战役进度。
- [ ] 所有新增规则均有说明、提示和移动端操作路径。
