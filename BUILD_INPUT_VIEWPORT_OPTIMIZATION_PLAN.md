# 建造、移动端操作与宽屏画面优化方案

> 文档状态：规划中  
> 版本：1.1  
> 更新日期：2026-07-24  
> 适用范围：`game/` H5 工程  
> 本文只确定产品与技术方案，不代表相关代码已经实现。

## 1. 文档目的

本文解决三类体验问题：

1. 移动端二次点击建造容易让塔影偏移，且在连战倒计时下增加无效点击；
2. 移动端拖动建造缺少边缘滚屏、明确的手势边界和完整的失败反馈；
3. 桌面端始终按 240×320 固定画面等比放大，宽屏左右空间无法用于显示更多战场。

本方案的首要约束是连战机制。连战阶段时间紧，移动端操作必须以“少点击、连续、可预期”为第一优先级，不能为了绝对防误触强制增加一次确认。

## 2. 最终产品决策

### 2.1 移动端建造

- 将“从塔卡直接拖入战场，松手立即建造”纳入首轮正式功能，作为连战时点击最少的快速路径。
- 保留“轻点塔卡进入预览，再拖动塔影建造”，作为容易理解且兼容当前习惯的标准路径。
- 取消移动端“再次点击当前格建造”。建造选位期间，普通轻点不会花费金币，也不会建造。
- 不增加“建造”确认按钮；画布外只保留“取消”作为建造阶段的必要退出操作。
- 合法位置松手立即完成建造；非法位置松手不扣钱、不退出选位，允许继续拖动。
- 拖动到战场边缘时自动平移地图，松手时仍以屏幕中的最终塔影位置结算。
- 桌面端的鼠标、键盘确认方式不受这项移动端规则影响。

### 2.2 其他移动端操作

- 普通战斗状态继续保持“轻点选择、单指拖图”。
- 建造选位状态改为“拖塔优先”，不让拖图和拖塔争夺同一根手指；远距离移动通过边缘滚屏完成。
- 升级、科技等可逆或低风险操作保持一次选择、一次执行；拆除等破坏性操作保留二次确认。
- 页面进入后台、锁屏或失去可见性时自动暂停战斗。
- 双指缩放不进入第一轮实现，避免在建造状态机稳定前增加新的手势冲突。

### 2.3 桌面宽屏

- 战斗画面保持纵向尺度不变，根据可用宽高比增加左右战场视野。
- 不拉伸瓦片，不把整个 240×320 画面缩小后塞进屏幕。
- 地图、城池、塔和敌人共同使用动态战场视口；UI 使用独立锚点布局。
- 标题、剧情、设置、结算等非战斗页面继续以经典 240×320 内容区居中显示。
- 提供“经典视野 / 宽屏视野”设置；桌面端默认宽屏，触屏竖屏默认经典视野。

## 3. 当前问题与原因

### 3.1 二次点击为什么容易偏移

当前移动端进入建造选位后，地图轻点执行以下判断：

1. 把触点换算成一个地图瓦片；
2. 将该瓦片与待建塔的内部锚点比较；
3. 相同则建造，不同则把锚点移动到触点瓦片。

问题在于塔的实际占地为 2×2 或 3×3，而内部建造坐标表示占地左上角。玩家看到的是完整塔影，通常会点击塔影中心；中心格并不等于左上角锚点，于是系统将玩家认为的“确认”解释成“移动”，塔影就会向右下或邻近位置偏移。

这不是单纯扩大点击区域就能彻底解决的问题。二次点击要求玩家理解不可见的内部锚点，本身不适合自由占地的触屏塔防。

此外，统一输入系统当前以 10 CSS 像素区分轻点和拖动。重复轻点时如果手指移动越过阈值，还可能从点击流程切换到拖动流程。拖动位置又包含建筑半占地和向上抬升量，因此两种路径的落点语义并不完全一致。

### 3.2 当前宽屏为什么只有黑边

当前渲染器始终使用 240×320 逻辑画面，并按窗口宽度和高度中较小的一项计算统一倍率。在 2048×1024 这类 2:1 屏幕中，画面填满高度后只占约 768 个 CSS 像素宽，剩余区域只能显示页面黑色背景。

战场裁剪、相机边界和实体剔除也依赖固定的 `MAP_VIEW_W = 240`，因此只修改 CSS 或扩大 Canvas 不会自动显示更多地图。

## 4. 移动端建造交互规格

### 4.1 标准流程

| 阶段 | 玩家操作 | 系统行为 |
| --- | --- | --- |
| 普通战斗 | 轻点可建地块 | 定位并打开建造栏 |
| 建造栏快速路径 | 按住塔卡并直接拖向战场 | 越过建造栏边界后收起栏，塔影接管同一个 pointer；合法位置松手立即建造 |
| 建造栏标准路径 | 轻点塔卡 | 关闭建造栏，进入塔影选位；塔影从已选地块开始 |
| 选位待机 | 轻点地图或塔影 | 不建造、不扣钱、不改变锚点；短暂提示“拖动塔影，松手建造” |
| 选位拖动 | 按住塔影并拖动 | 塔影跟随手指并吸附瓦片，实时显示合法性、占地和范围 |
| 合法位置 | 松手 | 立即建造一次、扣费一次、退出选位 |
| 非法位置 | 松手 | 不扣费、不退出；保留塔影并显示具体失败原因 |
| 任意选位阶段 | 点击取消或系统触发 `pointercancel` | 不建造，安全退出或保留到可恢复状态 |

移动端不再存在“点当前格确认”和“点其他格移动”的分支。这样轻点和拖动具有完全不同且稳定的含义：轻点不产生经济行为，只有明确拖动后的松手才能建造。

### 4.2 拖动意图判定

通用的 10 CSS 像素轻点阈值不应直接作为建造提交阈值。建造拖动需要独立状态和阈值：

- `pointerdown` 必须起始于塔卡，或塔影占地区域及其适度扩大的触摸热区；
- 建议第一版以 16 CSS 像素作为建造拖动意图阈值，并在真机上比较 12、16、20 三档；
- 未达到阈值的抬手始终视为轻点，不移动、不建造；
- 达到阈值后才进入 `placement-dragging`，随后松手才具有建造资格；
- 每个 pointer 序列最多触发一次建造；`pointercancel`、失去捕获或第二根手指加入都不能触发建造。

阈值必须以 CSS 像素计算，不能随 Canvas 高清倍率变化。2×2 和 3×3 建筑都要使用同一套手势判断。

### 4.3 塔影位置计算

塔卡选中时生成的初始塔影和拖动中的塔影应共用一个锚点约束入口；所有由手指位置驱动的移动则统一经过 `placementFromPointer`，避免以后再次出现两套落点语义。

拖动时继续保留当前“塔影显示在手指上方”的设计，避免手指遮挡合法性提示。换算时应同时考虑：

- 当前动态视口的地图原点；
- 相机位置；
- 塔的 2×2 / 3×3 占地中心；
- 手指上方的视觉抬升量；
- 地图边界和完整占地边界；
- 边缘滚屏过程中每一帧变化的相机位置。

用于显示的塔影锚点与最终提交给 `placeTower` 的锚点必须是同一个值。渲染层不得再次加入只影响视觉、不参与结算的隐藏偏移。

### 4.4 松手结算

合法位置松手后：

1. 使用当前塔影锚点执行一次合法性复核；
2. 成功后只扣除一次金币并创建一个塔；
3. 触发一次短震动和建造特效；
4. 退出建造选位并回到普通战斗。

非法位置松手后：

- 不扣金币，不创建塔，不关闭建造选位；
- 塔影停在最终吸附位置，允许玩家直接再次拖动；
- 显示明确原因，例如“空间不足”“必须朝向道路”“与已有建筑重叠”“金币不足”；
- 使用与成功明显不同的震动节奏，并允许在设置中关闭震动。

### 4.5 建造边缘滚屏

边缘滚屏只在 `placement-dragging` 状态启用，不能在单纯轻点或塔影待机时自行移动镜头。

建议参数：

- 战场四周设置约 20～24 逻辑像素的滚屏感应带；
- 手指进入感应带约 120ms 后开始移动，避免快速经过边缘时误触发；
- 越靠近边缘速度越快，初始建议从每秒 24 逻辑像素线性增加到每秒 96 逻辑像素；
- 上下边缘要排除画布外 HUD、建造栏和系统安全区，只以实际战场矩形计算；
- 相机到达地图边界后立即停止该方向移动；
- 相机移动的每一帧都重新计算塔影锚点，使塔影继续保持在手指上方；
- 松手时停止滚屏，并以滚屏后的最终世界坐标结算建造。

首版不加入惯性。建造需要精确落格，松手后镜头继续滑动会破坏可预期性。

### 4.6 塔卡直接拖入战场

“塔卡直拖”与基础拖动建造在首轮一起实现，但两者共用同一套塔影定位、合法性判断、边缘滚屏和松手结算，不能形成第三套建造坐标逻辑。

完整流程：

1. 玩家轻点可建地块打开建造栏；
2. 在塔卡上 `pointerdown`；
3. 手指未越过拖动阈值便抬起时，仍按普通轻点处理，进入塔影选位；
4. 手指越过阈值并离开建造栏、进入战场方向时，抑制塔卡的轻点事件；
5. 建造栏收起，所选塔影接管当前 pointer 并显示在手指上方；
6. 后续拖动、边缘滚屏和松手结算与普通塔影拖动完全一致；
7. 合法位置松手立即建造，非法位置松手则保留塔影选位，玩家可直接再次拖动。

轻点与拖动共存规则：

- 是否直拖由累计 CSS 移动距离和是否离开塔卡/建造栏热区共同决定；
- 手指只在卡片内部轻微滑动不能误触发建造；
- 一旦转入 `placement-dragging`，必须取消后续 `click`，避免松手后同时触发塔卡选择；
- 未解锁塔卡不能开始直拖；金币不足时允许查看红色预览，但松手不得扣费或退出选位；
- `pointercancel`、失去捕获、切后台或方向变化都不建造，并安全清理被收起的建造栏状态；
- 卡片直拖进入战场后的塔影位置必须经过同一个 `placementFromPointer`，不能使用卡片自身坐标推导落点。

该快速路径把“点塔卡，再拖塔影”合并为一次连续手势。连战中，从建造栏已经打开到完成建造只需一次按住、拖动和松手，不需要额外确认点击。

暂不默认开启连续建造同一种塔。连续扣费的误操作成本较高，应先观察连战实玩中是否确实频繁需要同塔连建，再决定是否增加“保持所选塔型”的可选快捷模式。

## 5. 其他移动端操作优化

### 5.1 优先级 P0

- 所有操作按上下文分流：普通战斗拖图、建造状态拖塔、文本页面滚动互不串线。
- 触摸热区不小于 44×44 CSS 像素；视觉图标可以保持原尺寸。
- 普通战斗轻点塔占地内任意格都能选中整座塔，不要求点击内部锚点。
- 拖图结束不能再触发地图轻点；`pointercancel` 不能产生确认行为。
- 页面 `visibilitychange` 为隐藏时自动暂停；恢复前不接受遗留 pointer 的抬手事件。

### 5.2 优先级 P1

- 画布外 HUD 根据当前上下文只展示有用按钮，避免同时出现互相竞争的入口。
- 建造状态显示塔名、价格和当前合法性；不显示额外“建造”按钮。
- 升级、装填、释放等操作使用大热区；拆除继续要求明确二次确认并使用危险色。
- 横屏时将状态和非上下文按钮放在战场两侧，尽量把中心区域留给地图。
- 设置中增加震动开关，并记忆横竖屏下的 UI 偏好。

### 5.3 暂缓项目

- 双指缩放；
- 带惯性的地图拖动；
- 双指拖图与单指拖塔同时操作；
- 可自由移动的悬浮按钮；
- 默认连续建造同一种塔。

这些功能并非永久排除，而是在基础手势状态机稳定前不应同时引入。

## 6. 桌面端动态宽屏规格

### 6.1 显示原则

宽屏模式类似固定纵向视野的策略游戏：

- 逻辑高度继续为 320；
- 地图可视高度继续以 276 为基线；
- 根据实际可用宽高比计算逻辑宽度；
- 新增宽度只用于显示左右更多世界和重新排布 HUD；
- TILE_SIZE 继续为 16，精灵、攻击范围和移动速度不随宽高比改变。

禁止直接把 240×320 拉伸到全屏。拉伸会破坏像素比例、圆形攻击范围和输入坐标，也不会真正增加地图信息。

### 6.2 建议计算方式

桌面宽屏以高度决定显示倍率，再由可用宽度反算逻辑宽度：

~~~text
heightScale = availableCssHeight / 320
rawLogicalWidth = availableCssWidth / heightScale
logicalWidth = floor(clamp(rawLogicalWidth, 240, 704) / 16) * 16
~~~

说明：

- 240 是经典最小宽度；
- 704 是现有最宽地图 44 格的像素宽度，可作为第一版视野上限；
- 向下取整到 16 的倍数可以让两侧瓦片边界稳定；
- 如果后续 HUD 需要更细腻的居中，可以允许 Canvas 宽度不按瓦片取整，但地图视口仍按完整格计算。

以 2048×1024 屏幕为例，纵向倍率约为 3.2，横向可得到约 640 逻辑像素，即约 40 格，而经典视野只能显示 15 格。

现有地图宽度为 20～44 格：多数关卡可以看到完整或接近完整的横向地图，最宽关卡仍保留少量横向移动。地图高度为 30～50 格，因此宽屏不会消除所有纵向移动。

### 6.3 地图小于视口时

当关卡地图宽度小于动态战场区域时：

- 地图整体在战场区域中水平居中；
- 相机 X 锁定在地图中央，不允许拖出空白世界；
- 地图以外区域使用与关卡色调匹配的暗色背景或 HUD 侧栏，不重复生成不存在的地图瓦片；
- 点击侧栏或地图外背景不能换算成合法地图格。

极宽屏超过 704 逻辑像素后的额外区域优先用于军情、选中塔详情、快捷说明等非遮挡 HUD，而不是无限扩大视野。

### 6.4 UI 布局

第一版宽屏布局遵循以下规则：

- 顶部资源、关卡、波次、速度和菜单按左右锚点排布，不再依赖固定 X 坐标；
- 底部建造栏保持可读宽度并水平居中，不横向拉伸原版装饰素材；
- 选中塔详情可放在底栏两侧新增空间；没有内容时保持克制，不堆叠永久面板；
- 军情面板移到地图边缘的安全区域，不能遮挡出生点、道路或建造落点；
- 暂停菜单、军议等模态面板按动态视口居中；
- 原版标题、剧情和结算页面绘制在居中的 240×320 经典内容矩形中，矩形外使用统一主题背景。

### 6.5 相机与窗口变化

窗口尺寸或方向变化时，应保持玩家当前观察的世界中心，而不是把镜头跳回左上角：

~~~text
oldWorldCenterX = cameraX + oldMapViewWidth / 2
newCameraX = clamp(
  oldWorldCenterX - newMapViewWidth / 2,
  0,
  mapWidthPx - newMapViewWidth
)
~~~

如果新视口已经宽于地图，则直接居中地图。建造拖动过程中发生 resize 或方向变化时，必须取消当前 pointer 手势，不能在坐标系变化后继续提交建造。

### 6.6 视野设置

建议提供：

- `经典视野`：战斗世界仍为 240 逻辑像素宽；
- `宽屏视野`：按上述规则动态扩展，桌面端默认；
- 触屏竖屏始终优先保证可点击尺寸，不为了多显示地图而缩小单位；
- 触屏横屏在完成宽屏架构后可以复用动态战场，但左右仍需为状态和操作按钮预留安全区。

如果未来加入联网排行或严格竞速比较，成绩记录应写入视野模式，以便必要时分组。当前单机玩法不需要因为宽屏信息优势限制视野。

## 7. 推荐技术结构

### 7.1 运行时视口模型

新增统一的运行时视口数据，避免继续从各系统直接读取固定常量：

~~~ts
interface ViewportMetrics {
  logicalWidth: number;
  logicalHeight: number;
  battleRect: { x: number; y: number; width: number; height: number };
  legacyRect: { x: number; y: number; width: 240; height: 320 };
  mapOriginX: number;
  cssScale: number;
  dpr: number;
}
~~~

职责建议：

- `Renderer` 计算并持有 `ViewportMetrics`；
- `Game` 根据输入模式、显示设置和当前页面选择经典或宽屏布局；
- `MapData` 使用运行时 `battleRect` 计算裁剪、相机边界和坐标转换；
- `EnemySystem`、`TowerSystem`、`CastleRenderer` 使用同一战场矩形做裁剪和剔除；
- `UI`、`BuildBar` 使用锚点布局，不再假定整个 Canvas 宽度等于 240；
- `InputSystem` 先从 CSS 坐标转换到动态逻辑坐标，再判断是否落在战场、经典内容区或画布外控件。

### 7.2 建造手势状态

不再只用全局 `isDragging` 表达所有操作，建议明确区分：

~~~ts
type PointerGesture =
  | 'idle'
  | 'tap-pending'
  | 'map-panning'
  | 'placement-pending'
  | 'placement-dragging'
  | 'text-scrolling';
~~~

输入系统需要向游戏层提供 pointer 起点、当前点、累计 CSS 距离、pointer 类型和取消原因。是否具有建造资格由 `placement-dragging` 状态决定，不能仅凭发生过一次 `pointermove` 判断。

### 7.3 主要影响文件

- `game/src/core/Input.ts`：建造拖动意图、pointer 状态、取消语义；
- `game/src/core/Game.ts`：输入上下文分发、建造状态转换、边缘滚屏协调；
- `game/src/core/MapData.ts`：动态视口、相机边界、地图居中、屏幕坐标转换；
- `game/src/core/Renderer.ts`：动态 Canvas 尺寸和 `ViewportMetrics`；
- `game/src/core/MobileControls.ts`：建造阶段隐藏确认按钮、保留取消和提示；
- `game/src/core/BuildBar.ts`：动态居中、塔卡按下/拖出判定和 pointer 交接；
- `game/src/core/UI.ts`：宽屏锚点布局；
- `game/src/core/Enemy.ts`、`Tower.ts`、`Castle.ts`：运行时裁剪范围；
- `game/index.html`：桌面宽屏容器、主题背景和触屏安全区。

## 8. 实施阶段

### 阶段 A：建造输入基线与回归测试

- 为当前二次点击偏移建立 2×2、3×3 回归测试；
- 建立轻点、拖动、非法落点、`pointercancel`、重复 `pointerup` 测试；
- 记录当前真机拖动手感，作为修改后的对照基线。

完成标准：测试能稳定复现“点击塔影中心导致锚点移动”，且不会依赖真实浏览器随机时序。

### 阶段 B：移动端拖动建造状态机

- 移除移动端二次点击建造和点击移动塔影；
- 建立独立的建造拖动意图阈值；
- 实现塔卡轻点与“直接拖入战场”的互斥判定，并把同一 pointer 交给塔影；
- 合法松手立即建造，非法松手保留选位；
- 建造阶段移除画布外“建造”按钮，仅保留取消；
- 统一塔影显示与提交坐标。

完成标准：建造栏打开后可用一次连续的塔卡拖动完成建造；轻点塔卡仍可进入预览；任何地图轻点都不会扣费、移动塔影或创建建筑。

### 阶段 C：边缘滚屏与移动端操作收口

- 实现四边渐进滚屏；
- 增加失败原因、震动设置、后台自动暂停；
- 完成手机竖屏、横屏和不同 DPR 的真机测试；
- 根据实测调整拖动阈值、感应带和滚屏速度。

完成标准：玩家可以在不退出建造状态、不额外点击按钮的情况下，把塔拖到当前视口外的地图区域并完成建造。

### 阶段 D：动态战场视口

- 引入 `ViewportMetrics`；
- 先改 Renderer、MapData 和坐标转换；
- 再迁移敌人、塔、城池和特效的裁剪范围；
- 保持非战斗页面经典内容区不变。

完成标准：16:9 和 21:9 下能显示更多横向地图，单位大小不变，地图边界、点击格和攻击范围保持正确。

### 阶段 E：宽屏 UI 与全关卡验收

- 改造顶部栏、建造栏、军情和模态面板锚点；
- 增加经典/宽屏设置；
- 验收 9 张地图和窗口实时 resize；
- 验证宽屏和经典视野下塔卡直拖的热区与坐标交接一致。

完成标准：宽屏不再出现截图中的大面积无用途黑边；地图较窄时居中且两侧空间有一致的视觉处理。

## 10. 验收标准

### 10.1 移动端

- 建造选位时，轻点任何地图位置都不会建造或移动塔影；
- 玩家通过一次明确拖动并松手即可完成合法建造，不需要点击确认按钮；
- 建造栏打开后，玩家可以从塔卡开始一次连续拖动并完成建造，中途不需要先点选塔卡；
- 轻点塔卡仍能稳定进入塔影预览，轻微手抖不会误转成塔卡直拖；
- 点击塔影中心不会再导致 2×2 / 3×3 锚点偏移；
- 非法松手不扣费、不退出，错误原因清楚；
- 边缘滚屏可到达完整地图，滚动期间塔影不跳格、不脱离手指；
- 连战倒计时期间从选塔到完成建造的操作次数不高于当前拖动建造流程；
- 取消、来电、切后台和 pointer 中断都不会误建。

### 10.2 桌面端

- 16:9 和更宽屏幕能够显示比经典模式更多的左右地图；
- 所有瓦片保持正方形，单位和文字不因宽屏被横向拉伸；
- 2048×1024 下战场可达到约 640 逻辑像素宽；
- 地图较窄时居中，地图外区域不可点击；
- resize 前后的世界中心尽量保持不变；
- 经典视野仍可使用，存档、战斗逻辑和数值不因视野变化而改变。

## 11. 明确不做的事情

- 不在移动端增加强制“建造”确认按钮；
- 不保留移动端二次点击当前格建造作为默认入口；
- 不用缩小所有元素的方式换取更多地图面积；
- 不把 240×320 画面非等比拉伸至全屏；
- 不让视野宽度参与攻击、寻路、刷怪或计时逻辑；
- 不在第一轮同时实现双指缩放、惯性拖图和连续建造；
- 不因宽屏改造重做原版地图或修改塔的占地数据。

## 12. 实施前需要保留的实玩基线

正式改动前，至少保存以下短视频或操作记录：

- 当前拖动建造一座 2×2 塔和一座 3×3 塔；
- 二次点击塔影中心发生偏移的案例；
- 连战倒计时中连续完成三次建造；
- 当前手机竖屏、横屏和 2048×1024 桌面宽屏画面。

后续每个阶段都与这些基线对照。目标不是重新发明整套操作，而是在保留现有拖动建造速度和手感的前提下，删除产生歧义的点击路径，并把空闲屏幕空间真正转化为可用战场视野。
