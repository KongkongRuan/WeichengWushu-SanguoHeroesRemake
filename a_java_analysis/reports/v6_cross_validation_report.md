## 一、本轮发现并修复的问题

### 1.1 模块03: `hideNotify()` 状态检查条件错误 (★ 严重)

**文件**: `03_callbacks.java`  
**VF行号**: L27115-27166

**问题**: `hideNotify()` 的状态检查条件包含了不应该包含的状态值。

**VF嵌套if结构分析**:
```
if (l != 46) {          // l==46 → 不进入, 直接返回
    if (l != 48) {      // l==48 → 不进入, 直接返回
        if (l != 47) {  // l==47 → 不进入, 直接返回
            if (l != 2) {
                if (l != 12) {
                    ...
                    if (l != 23) {
                        return;  // l不是以上任何值
                    }
                }
            }
            // 只有 l∈{2,12,13,22,23} 才到达这里
            this.a(3);
            ...
        }
    }
}
```

**正确逻辑**:
- `l ∈ {2, 12, 13, 22, 23}` → 执行 `a(3)` 等操作
- `l ∈ {46, 47, 48}` → **不执行任何操作** (直接返回)
- 其他值 → return

**原错误代码**:
```java
if (this.l != 46 && this.l != 48 && this.l != 47 
    && this.l != 2 && this.l != 12 && this.l != 13 
    && this.l != 22 && this.l != 23) {
    return;
}
this.a(3);  // ← 46,48,47也会执行到这里!
```

**修复后代码**:
```java
if (this.l != 2 && this.l != 12 && this.l != 13 
    && this.l != 22 && this.l != 23) {
    return;
}
this.a(3);  // ← 只有2,12,13,22,23才会执行
```

**影响**: 当游戏处于状态46/47/48（结局动画相关状态）时，如果Canvas被隐藏（如来电中断），原错误代码会不正确地执行 `a(3)`（菜单跳转），导致游戏状态被错误修改。

## 三、已知的设计性差异 (不影响功能)

### 3.1 paint()/keyPressed() 的 try-catch 简化
VF中每个字段访问都有独立的try-catch块（混淆器添加的异常处理表），模块中简化为一个大try-catch。由于每个独立catch都是 `printStackTrace() + return`，简化版本在功能上等价。

### 3.2 run() 中 b1018 检查的位置
字节码中 `b1018` 检查(offset 36-3c)在try范围(41-116)之外。模块将其放在try块内。如果 `CMidlet.a()` 抛出异常，原始字节码不会捕获，但模块会捕获。实际影响极小，因为 `CMidlet.a()` 通常不抛出异常。

### 3.3 VF的 int-as-boolean 问题
VF中多处 `if (var5)` 其中var5是int类型（实际字节码是`ifne`），模块保留了VF的写法。
