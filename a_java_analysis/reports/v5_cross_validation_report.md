## 一、重大发现: 混淆器造成的方法签名冲突

### 1.1 问题描述

通过 javap 字节码分析, 发现原始 `a.class` 中存在**仅返回类型不同**的方法重载。
这在 Java 源码层面是非法的 (Java 不允许仅通过返回类型区分重载),
但在 JVM 字节码层面是合法的 (字节码方法签名包含返回类型)。

这是混淆器 (如 ProGuard/zkm) 对原始代码进行重命名后产生的副作用。

### 1.3 对 H5 重制的影响

1. **模块化拆分**: 这些冲突方法被分散到不同模块 (06/07) 中, 每个模块只包含一种返回类型的版本

## 二、已修复的问题

### 2.1 模块06: `a(int,int,int)` 角度计算 — 二分查找逻辑错误 (★ 严重)

**文件**: `06_logic_a_z.java`  
**问题**: 二分查找中 `lo = this.j1049[mid]` 将 lo 错误赋值为数组值而非索引, 导致查找逻辑完全错误  
**影响**: 游戏中所有角度计算 (塔射击方向、敌人移动方向) 都会出错  
**修复**: 按照 VF L10009-10072 重写二分查找逻辑:
  - `mid = (hi + lo) >> 1` 计算中点
  - `j1049[mid] > absN` → `hi = mid - 1`
  - `j1049[mid] >= absN` → break (找到)
  - 否则 → `lo = mid + 1`

### 2.2 模块06: `b(int)` 瓦片可建造检查 — 漏掉 `m(var1)` 调用 (★ 严重)

**文件**: `06_logic_a_z.java`  
**问题**: 当瓦片不可建造时, 应该调用 `this.m(var1)` 方法 (VF L16862), 但模块中漏掉了  
**影响**: 瓦片不可建造时缺少反馈逻辑, 可能导致游戏状态不一致  
**修复**: 在 `return false` 前添加 `this.m(var1)` 调用  
**同时修复**: 变量类型从 `byte` 改为 `boolean` (c(int) 返回 boolean)

### 2.3 模块06: 遗漏 `boolean b(int,int)` 方法 (★ 重要)

**文件**: `06_logic_a_z.java`  
**问题**: VF L16868-16889 有 `boolean b(int,int)` 瓦片可建造检查(带坐标), 但模块中遗漏  
**影响**: 带坐标的瓦片可建造检查功能缺失  
**修复**: 新增 `boolean b(int,int)` 方法, 逻辑为:
  1. `a(var1,var2)` 计算瓦片索引
  2. `c(var3)` 检查奇偶性
  3. 检查 `E1163[var3] < 8` 和 `D1162[var3] < 6`
  4. 不可建造时调用 `m(var3)`

### 2.4 模块08: `d()` 方法 — `if(url!=null)` 逻辑错误 (★ 中等)

**文件**: `08_bytecode_recovered.java`  
**问题**: `if (url != null)` 分支体为空, 什么都不做, 逻辑错误  
**正确逻辑**: 应该是 `if (url == null)` 时使用 `f1186`  
**修复**: 将条件改为 `if (url == null)`, 与模块06和字节码分析一致

## 三、已知但无法修复的结构性问题

### 3.1 方法签名冲突 (见第一节)

由于混淆器造成了仅返回类型不同的方法重载, 这些冲突在 Java 源码层面无法解决。
模块化拆分通过将冲突方法分散到不同模块来"隐藏"此问题。

### 3.2 大量方法体省略

模块06和07中, 对于过长的方法 (如 `a(int,int)` 精灵图加载有257行), 只保留了方法签名和注释, 
方法体用 `// ... 完整见 VF L...` 代替。这是为了避免编辑器性能问题而做的设计决定。

### 3.3 VF 反编译的 int-as-boolean 问题

VF 中多处出现 `if (var5)` 其中 var5 是 int 类型, 这在标准 Java 中不合法。
实际字节码是 `ifne` (if not zero), 应为 `if (var5 != 0)`。模块中保留了 VF 的写法。
