## 1. 概述

`a.java` 是游戏的核心类,继承自 `com.nokia.mid.ui.FullCanvas` 并实现 `Runnable`。
这是 J2ME 时代的典型游戏架构:一个主类承载全部游戏逻辑。

```
public final class a extends FullCanvas implements Runnable
```

- **原始字节码**: `weichengwushuang_j2me.jar` → `a.class` (166308 字节, Java 5)
- **主要反编译工具**: Vineflower 1.12.0 (成功率最高, 仅 5 个方法失败)
- **辅助验证工具**: CFR 0.152, Procyon 0.6.0, javap (JDK 17)
- **代码规模**: Vineflower 输出 27524 行
- **字段数量**: 约 250+ 个（含大量混淆名）
- **方法数量**: 约 180+ 个（含大量重载,方法名从 `A()` 到 `z()`）
- **构造函数**: `public a()` 位于第 4741 行,opcode 数 20927,包含所有硬编码数值

### 1.1 反编译质量

CFR 反编译器在摘要中标记了大量问题:
- `Unable to fully structure code` — 控制流无法完全重建
- `Could not resolve type clashes` — 类型推断冲突
- `Loose catch block` — 异常处理结构松散
- `Back jump on a try block` — try 块回跳（字节码混淆特征）
- `Opcode count of 20927 triggered aggressive code reduction` — 构造函数过大,触发激进代码缩减

这意味着 **部分方法体可能不完全准确**,但字段声明、构造函数中的数值初始化、方法签名是可靠的。

## 7. 反编译问题与限制

### 7.1 CFR 报告的问题

| 问题 | 影响 | 出现位置 |
|------|------|----------|
| `Unable to fully structure code` | 控制流可能不完全准确 | 多个方法 |
| `Could not resolve type clashes` | 类型推断可能有误 | `R()`, `X()`, `a(int,int,boolean)` 等 |
| `Loose catch block` | 异常处理结构松散 | `g(int)` |
| `Back jump on a try block` | try块异常(混淆特征) | `d()` |
| `Opcode count 20927 aggressive reduction` | 构造函数被激进缩减 | `a()` 构造函数 |

### 7.2 混淆特征

1. **字段名混淆**: `a1001`, `b1015`, `c1107` 等 — 字母+编号格式
2. **方法名混淆**: `a()`, `b()`, `A()`, `B()` 等 — 单字母,大量重载
3. **变量名混淆**: `n`, `n2`, `n3`, `object`, `object2` 等 — 通用名
4. **类型混淆**: `(int)"string"` 强制转换 — CFR 的类型推断产物
5. **数组别名**: `nArray`/`nArray2` 指向同一对象 — CFR 的反编译产物

### 7.3 可靠性评估

| 内容 | 可靠性 | 说明 |
|------|--------|------|
| 字段声明 | ★★★★★ | 完全可靠,直接来自类定义 |
| 方法签名 | ★★★★★ | 完全可靠,直接来自方法声明 |
| 构造函数数值 | ★★★★☆ | 基本可靠,但激进缩减可能遗漏部分 |
| 方法体逻辑 | ★★★☆☆ | 部分方法控制流可能不准确 |
| 字符串内容 | ★★★★★ | 完全可靠,Unicode转义可准确解码 |

## 附录: 构造函数中的局部变量映射

构造函数 `a()` 开头定义了以下局部变量常量:

```java
int n2 = 4;   // 数组大小常量
int n3 = 1;   // 索引1
int n4 = 3;   // 索引3
int n5 = 2;   // 索引2
```

这些常量在数组初始化中用作索引和大小,例如:
- `object[n3] = ...` 等价于 `object[1] = ...`
- `new byte[n2]` 等价于 `new byte[4]`
