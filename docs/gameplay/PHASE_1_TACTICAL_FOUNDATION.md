# 第一阶段方案：战术基础与局内决策

> 文档状态：已实现，待完整实玩验收
> 依赖：[玩法增强总纲](../../GAMEPLAY_ENHANCEMENT_ROADMAP.md)
> 预计开发量：约 1,800～3,000 行新增或重构代码
> 预计人工量：7～12 人日
> Token 区间：26 万～42 万
> 建议阶段预算：35 万 Token

## 1. 阶段目标

第一阶段负责搭建后续玩法共用的强化规则基础，并交付一个已经能够重复游玩的最小强化模式。完成后，玩家应当可以：

- 在经典和强化规则之间明确切换；
- 在出兵前看到下一波主要军情；
- 通过快速出兵积累连战奖励；
- 在固定波次结束后进行军议三选一；
- 利用四组增强机关联动改变布局；
- 在结算和战役地图查看通关、无人入城、及时出兵勋章；
- 查看本局主要建筑的伤害、击杀和联动贡献。

## 2. 本阶段非目标

- 不实现主动武将技能。
- 不新增敌方名将主动机制。
- 不实现历史地图事件、无尽或每日挑战。
- 不改变经典规则下现有火冰互斥、滚油遇火和沸水灭火逻辑。
- 不新增联网服务。
- 不把 1～3 倍速列为新功能；现有倍速只做回归验收。

## 3. 总体技术设计

### 3.1 最小模块边界

推荐新增以下模块，名称可在实现前小幅调整：

~~~text
game/src/enhancement/
├── Ruleset.ts              # classic/enhanced 与功能开关
├── SeededRng.ts            # 可保存、可复现的随机数
├── RunState.ts             # 强化对局状态
├── CombatEvents.ts         # 最小战斗事件总线与类型
├── WaveIntel.ts            # 波次计划、预告、连战奖励
├── BattleRecords.ts        # 勋章和最佳记录
├── CouncilSystem.ts        # 军议抽取、选择和叠加
├── ModifierResolver.ts     # 统一数值修正入口
├── ReactionSystem.ts       # 强化联动解析
└── BattleStats.ts          # 每塔与整局统计

game/src/data/enhancement/
├── councilData.ts
└── reactionData.ts
~~~

Game.ts 只负责创建系统、状态切换和把事件连接到 UI。Enemy.ts 和 Tower.ts 只增加必要查询或事件出口，不在其中维护军议选择列表和勋章规则。

### 3.2 强化运行状态

建议使用显式、可序列化的状态：

~~~ts
export interface EnhancedRunStateV1 {
  version: 1;
  contentVersion: number;
  seed: number;
  rngState: number;
  currentWavePlan: WavePlan | null;
  deployment: DeploymentState;
  council: CouncilRunState;
  reactions: ReactionRunState;
  medalProgress: MedalProgress;
  stats: SerializedBattleStats;
}
~~~

不要保存闭包、DOM 引用、图片对象或类实例。存档加载后由各系统从纯数据恢复。

### 3.3 修正器顺序

所有增强数值遵循固定结算顺序：

1. 原版基础值；
2. 原版科技与武将被动；
3. 阵营或模式修正；
4. 军议加法修正；
5. 军议乘法修正；
6. 临时状态修正；
7. 下限、上限和整数取整。

第一阶段虽然还没有阵营政令，也要预留第 3 层接口。经典规则直接在第 2 层后返回原版结果。

### 4.2 存档建议

战斗存档增加可选字段：

~~~ts
rulesetId?: RulesetId;
enhancedRun?: EnhancedRunStateV1;
~~~

另建强化档案键，用于战役勋章和最佳记录。现有军令档案继续负责军令发放、正攻/奇谋和救援，不把两种职责强行合并。

- 旧存档缺少 rulesetId 时按 classic 加载。
- 旧设置缺少规则字段时按 classic 加载。

- 新勋章档案不重复发放旧奖励；历史数据只能做展示迁移或标记为全局已完成。

## 5. 里程碑 P1.1：波次情报与连战奖励

> 状态：已实现（时间与数值指标待实玩）
> 建议预算：5 万 Token
> 合理区间：3.5 万～5.5 万 Token

### 5.1 波次计划

把“决定下一波内容”和“正式开始下一波”分离。强化规则在波次可开始时创建并缓存 WavePlan，预告和正式生成共同读取同一份计划，避免预告内容与实际内容不一致。

~~~ts
export interface WavePlan {
  wave: number;
  enemyType: number;
  count: number;
  spawnLane: number | null;
  bossType: number;
  bossId: number;
  elite: boolean;
}
~~~

如果经典规则的原版随机流程不适合预计算，则只在 enhanced 分支使用计划对象，classic 继续走当前 computeWaveParams 行为。

### 5.2 UI 信息

波次准备期间显示：

- 第几波/总波数；
- 主要兵种图标和数量；
- 可确定时显示出生路线；
- 名将波显示名字、头像和“详情将在第二阶段开放”的基础免疫说明；
- 连战等级和本波可获得的额外金币。

240×320 画布只显示一行摘要；详细信息通过点击/触摸展开，不能遮挡建造区域。

### 5.3 连战规则初稿

- 第一波不计时，也不提供连战金币。
- 从第二波开始，在场上清空且可出兵时启动准备计时；暂停和后台不可见时间不计入。
- 2 秒内出兵：连战 +2。
- 2～5 秒出兵：连战 +1。
- 超过 5 秒：连战归零，但仍可正常出兵。
- 已有连战且下一波可出兵时，HUD 与触屏出兵按钮显示剩余时间；出兵或超时后立即隐藏。
- 连战上限 5。
- 波次完成时奖励 连战 × 2 金，单波最多 10 金。
- 使用清场军令完成当前波时，本波不发连战金币。
- 现有“全波次及时出兵”仍以每波不超过 5 秒为条件。

以上数值为第一轮实玩基线，完成 P1.5 后允许调整，但修改必须集中在数据配置中。

### 5.4 验收标准

- [x] 预告兵种、数量、路线和名将与实际生成一致。
- [x] 读档后当前 WavePlan 不会重新随机。
- [ ] 暂停、切后台和剧情界面不消耗准备时间（代码与自动化测试已完成，待在线手工复核）。
- [ ] 1、2、3 倍速不改变以真实可操作时间定义的准备窗口。
- [x] 连战奖励只发放一次并计入统计。
- [x] 经典规则下没有预告奖励且经济完全不变。

## 6. 里程碑 P1.2：勋章、结算与战役记录

> 状态：已实现（完整通关展示待实玩）
> 建议预算：3 万 Token
> 合理区间：2.5 万～4 万 Token

### 6.1 三枚勋章

每个规则集、阵营和战役分别记录：

1. **破城**：完成关卡。
2. **无漏**：没有敌人进入我城，且未使用军令。
3. **迅捷**：第二波起每波均在 5 秒内出兵，且未使用军令。

军令辅助通关仍记录“已通关”和奇谋胜利，但不授予无漏、迅捷和正式挑战成绩。现有军令首次奖励逻辑保持原样，不能因展示勋章而重复发放。

### 6.2 记录键

建议记录键包含：

~~~text
规则集 / 游戏模式 / 阵营 / 战役编号
~~~

自由模式和王者之路即使使用同一地图也分开记录。后续每日挑战和无尽使用独立记录类型，不复用战役键。

### 6.3 展示

- 关卡结算显示本局获得、此前已获得和未满足原因。
- 战役地图的战役名称旁显示 0～3 枚小型勋章。
- 存档摘要显示当前规则集。
- 使用军令时提前提示本局哪些成绩不再有效。

### 6.4 验收标准

- [x] 同一关重复通关不重复发军令。
- [x] 三国同一地图的勋章可以分别记录。
- [x] 经典、强化、自由模式记录不会互相覆盖。
- [x] 读档后漏兵和及时出兵条件保持一致。
- [x] 结算能解释勋章失败原因。

## 8. 里程碑 P1.4：增强机关联动

> 状态：已实现（3 倍速反馈待实玩）
> 建议预算：9 万 Token
> 合理区间：6 万～10 万 Token

### 8.1 设计原则

现有代码已经实现原版异常状态和部分联动。本里程碑不替换原版逻辑，而是在 enhanced 分支处理额外 ReactionResult。每次联动都必须：

- 有明确的触发者和目标；
- 有防止单次攻击重复触发的事件 ID；
- 有短文本、图形和伤害数字反馈；
- 记录贡献来源；
- 遵守 Boss 免疫，不通过联动绕过原版完全免疫；
- 具备每目标冷却或消耗规则，避免高频无限触发。

### 8.2 第一版四组联动

#### A. 油火引燃

- 沿用现有“滚油实际遇到燃烧敌人才引燃”的判定。
- 强化规则额外在受影响道路格生成短暂燃烧地带。
- 燃烧地带持续 16 个逻辑帧，每 4 帧对格内敌人造成 4 点火焰伤害。
- 同格刷新持续时间但不叠加多个伤害实例。
- 火焰免疫敌人不受伤，但仍可触发视觉。

#### B. 冰火爆裂

- 火焰命中冻结目标，或寒冰命中燃烧目标时触发。
- 清除目标当前火焰和冻结状态。
- 对目标及相邻小范围造成 20 + 4 × 建筑等级 的伤害。
- 单个目标 30 个逻辑帧内最多触发一次。
- 范围伤害仍按各敌人防御和免疫结算。

#### C. 石灰沸灼

- 沸水命中处于中毒状态的敌人时触发。
- 消耗最多 16 帧剩余中毒时间。
- 施加 30 帧破防，防御降低 2 + 建筑等级的一半向下取整。
- 防御最低按 0 结算，不产生负防御增伤。
- 同目标仅刷新持续时间，不叠加破防数值。

#### D. 麻痹穿刺

- 突刺或擂木命中麻痹目标时触发。
- 本次原始伤害提高 25%。
- 消耗 10 帧麻痹时间，最低降至 0。
- 一次多段攻击只在首次有效命中时获得加成。

### 8.3 数据与状态

新增状态尽量使用单独字段，不复用 Enemy.effect 单值。Enemy.effect 目前只是兼容显示优先级，不能成为强化联动的唯一事实来源。

建议 ReactionResult 至少包含：

~~~ts
interface ReactionResult {
  reactionId: string;
  sourceTowerId: number;
  targetEnemyId: number;
  damage: number;
  consumedStatuses: string[];
  appliedStatuses: string[];
  visualCue: string;
}
~~~

### 8.4 验收标准

- [x] 四种联动在强化规则中可稳定触发。
- [x] 经典规则逐项保持现有测试结果。
- [x] 多段、范围和持续伤害不会重复结算同一事件。
- [x] Boss 免疫与增强联动的关系均有测试。
- [ ] 3 倍速下反馈仍可辨认且不会漏结算。
- [x] 联动伤害能归属到正确建筑和战报。

## 9. 里程碑 P1.5：战报、体验打磨与阶段验收

> 状态：代码已实现，待完整实玩与数值验收
> 建议预算：6 万 Token
> 合理区间：4 万～6.5 万 Token

### 9.1 战斗统计

为每座建筑分配稳定 instanceId，至少记录：

- 直接伤害；
- 持续伤害；
- 联动伤害；
- 击杀数；
- 触发联动次数；
- 有效控制帧数；
- 建造和升级总投入。

选中塔时在详情区显示简要数据；关卡结算显示伤害前三、联动次数、最高连战和额外金币。统计只用于展示和后续计分，不反向修改战斗。

### 9.2 事件归属

- 持续伤害必须保存来源建筑 ID，不能全部归为环境伤害。
- 地面燃烧由首次创建或最近刷新者归属，规则需固定。
- 联动有两个参与塔时，伤害归触发塔，辅助塔获得“助攻联动”计数。
- 塔被拆除后保留本局统计，结算显示“已拆除”。

### 9.3 实玩矩阵

至少完成以下手工回归：

| 维度 | 覆盖 |
|---|---|
| 规则集 | 经典、强化 |
| 模式 | 王者之路、已解锁自由模式 |
| 阵营 | 蜀、魏、吴 |
| 地图 | 黄巾、赤壁、夷陵至少 3 张 |
| 输入 | 键鼠、触屏 |
| 速度 | 1x、2x、3x |
| 存档 | 新局、旧 v1 迁移、波间保存恢复 |
| 成绩 | 正攻、奇谋、漏兵、快速、延迟出兵 |

### 9.4 数值验收指标

第一轮目标不是绝对平衡，而是排除明显失控：

- 连战奖励不应占正常关卡总收入的 15% 以上；
- 任一军议不应成为所有阵营、所有地图的无条件最优；
- 单一联动不应贡献总伤害的 50% 以上；
- 强化规则平均通关时间不比经典增加 30% 以上；
- 军议界面平均决策时间目标为 5～15 秒；
- 低端移动端 3 倍速和高敌人数场景仍可完成建造操作。
