# 第三阶段方案：历史事件与重玩模式

> 文档状态：概要完成，待第二阶段实玩后细化
> 依赖：[玩法增强总纲](../../GAMEPLAY_ENHANCEMENT_ROADMAP.md)
> 前置方案：[第一阶段](PHASE_1_TACTICAL_FOUNDATION.md)、[第二阶段](PHASE_2_HEROES_AND_BOSSES.md)
> 预计开发量：约 4,000～7,000 行新增或重构代码
> 预计人工量：22～40 人日
> Token 区间：70 万～125 万
> 建议阶段预算：95 万 Token

## 1. 阶段目标

第三阶段解决“内容通关后为什么还要继续玩”的问题。通过数据化地图事件、无尽模式、每日挑战和统一计分，让 9 张现有地图在不同规则下产生长期重玩价值。

完成后应交付：

- 一套不依赖硬编码关卡编号的事件运行框架；
- 9 张战役地图各至少一个专属事件或次要目标；
- 可持续生成波次并支持中断恢复的无尽模式；
- 使用固定种子的本地每日挑战；
- 挑战限制、统一计分、最佳记录和版本隔离；
- 200 波无尽稳定性测试和每日挑战确定性测试。

## 2. 开始条件

- 第一阶段的种子随机、运行存档、战斗事件和统计接口已经稳定。
- 第二阶段的将技、Boss 特性和阵营政令完成组合回归。
- 经典模式保护测试持续通过。
- 增强规则至少完成三国各两图实玩。
- 内容数据结构已经与原版 gameData 分离。
- 当前性能基线已记录，包括 80 敌人附近、3 倍速和移动端帧率。

## 3. 本阶段非目标

- 不实现联网排行榜、反作弊或账号系统。
- 不保证修改系统时间后仍能防止重复进入其他日期挑战。
- 不制作地图编辑器或玩家创意工坊。
- 不在运行时任意重写地图寻路拓扑；需要改路的事件先使用已有合法路线和出生点。
- 不新增大量地图；优先提高现有 9 图利用率。
- 不承诺无限数值增长，超过设计上限后允许进入封顶循环。

### 4.2 事件定义

事件使用白名单触发器和动作，不允许在数据文件中存放任意可执行脚本。

~~~ts
interface LevelEventDefinition {
  id: string;
  levelId: number;
  enabledModes: string[];
  initialState: Record<string, number | boolean | string>;
  objectives: EventObjectiveDefinition[];
  rules: EventRuleDefinition[];
}

interface EventRuleDefinition {
  id: string;
  once: boolean;
  trigger: EventTriggerDefinition;
  conditions: EventConditionDefinition[];
  actions: EventActionDefinition[];
}
~~~

第一版触发器至少支持：

- battle_start；
- wave_ready、wave_start、wave_complete；
- boss_spawn、boss_health_threshold、boss_defeated；
- logic_timer；
- objective_health_threshold、objective_completed、objective_failed；
- player_event_action；
- lives_threshold。

第一版动作至少支持：

- 显示预告或剧情短句；
- 添加、移除或限时应用修正器；
- 在合法出生点生成敌人或事件实体；
- 创建区域、目标物或交互按钮；
- 对事件实体造成伤害或治疗；
- 发放金币、分数、勋章进度；
- 安排延迟动作；
- 标记目标成功或失败。

### 4.3 运行状态

事件状态必须可完整保存：

- 已触发 ruleId；
- 活动计时器；
- 目标状态、生命值和位置；
- 事件实体；
- 当前环境修正；
- 玩家已使用的事件动作和剩余次数；
- 奖励是否发放；
- 内容版本和随机状态。

加载时不重复播放已经结算的奖励，不重复触发一次性规则。

## 5. 里程碑 P3.0：事件框架与编辑约束

> 状态：待第二阶段完成后开发
> 建议预算：15 万 Token
> 合理区间：10 万～18 万 Token

### 5.1 工作内容

- 实现 LevelEventSystem、触发器、条件和动作白名单。
- 实现区域、静态目标物和移动事件实体三种基础对象。
- 建立事件层渲染、点击、目标血条和提示优先级。
- 接入 CombatEvents、ModifierResolver、SeededRng 和 RunState。
- 为 levelEventData 增加启动时配置校验。
- 建立坐标检查工具，验证事件坐标位于地图范围和合法层。
- 实现事件存档往返和一次性奖励保护。

### 5.2 配置校验

启动或构建测试时至少检查：

- 事件、规则和目标 ID 唯一；
- 引用的地图、出生点、武将、修正器和动作存在；
- 坐标在地图范围内；
- 触发规则不存在直接自循环；
- once 规则不会在加载时重复；
- 奖励动作具有唯一奖励键；
- 移动实体使用已验证路线，不自行猜测方向链。

### 5.3 性能约束

- 事件条件由相关战斗事件触发，不每个逻辑帧遍历全部规则。
- logic_timer 使用统一调度队列。
- 静态区域预计算格子集合。
- 事件日志使用固定长度环形缓冲，只保存诊断所需记录。

### 5.4 验收标准

- [ ] 三种基础事件对象均可通过纯配置创建。
- [ ] 每类触发器和动作至少有一个测试。
- [ ] 非法配置在开发构建中给出明确错误。
- [ ] 存档恢复后计时、目标和奖励状态一致。
- [ ] classic 规则不加载事件定义。

## 6. 里程碑 P3.1：三张试点历史事件地图

> 状态：设计草案
> 建议预算：16 万 Token
> 合理区间：12 万～20 万 Token

试点应覆盖三种不同能力：环境周期、玩家机关、移动保护目标。具体坐标必须在开发前根据 level JSON 和实际画面标定。

### 6.1 赤壁之战：东风

- 每两波预告并改变一次风向。
- 风向影响强化规则燃烧地带的扩展方向和持续时间，不改变经典火焰。
- 战前和 HUD 显示当前/下次风向。
- 周瑜或诸葛亮将技可以短暂锁定有利风向，但不能永久覆盖事件。
- 次要目标：在三个不同风向下分别触发一次油火引燃。

验证能力：周期环境修正、将技覆盖优先级、联动统计。

### 6.2 攻克成都：山道落石

- 地图上配置若干落石触发点，每场各可使用一次。
- 玩家点击后显示落区，再次确认后延迟落石。
- 落石对普通敌人造成高额物理伤害，对名将造成有限伤害和短打断。
- 使用落石不算军令，但会影响对应次要目标的“纯机关通关”评价。
- 落石不能堵塞路线，也不生成永久障碍。

验证能力：玩家事件动作、区域预告、次数保存、Boss 伤害边界。

### 6.3 长坂坡：护送百姓

- 在预设安全路线生成少量移动百姓，不占用原版敌人池。
- 百姓只在安全路径节点之间移动，不参与攻击和寻路竞争。
- 附近有敌人时停下并显示危险标记。
- 玩家需要在限定波次前让一定数量抵达我城附近安全区。
- 失败不直接导致战斗失败，但失去事件奖励和完美事件标记。

验证能力：移动事件实体、保护目标、成功/失败分支和独立统计。

### 6.4 试点验收

- [ ] 三个事件完全由配置和通用动作组合完成；确需专用动作时记录理由。
- [ ] 每个事件都有开始预告、当前状态和结算结果。
- [ ] 读档不会重置风向、落石次数或百姓位置。
- [ ] 军议、将技和 Boss 特性与事件组合没有软锁。
- [ ] 经典规则三图行为保持不变。

## 7. 里程碑 P3.2：其余六张历史事件地图

> 状态：概念草案，待试点反馈
> 建议预算：18 万 Token
> 合理区间：12 万～25 万 Token

### 7.1 事件概念

| 战役 | 事件草案 | 主要决策 |
|---|---|---|
| 黄巾之乱 | 妖术祭旗 | 优先摧毁强化普通兵的祭旗，还是继续积累经济 |
| 虎牢关 | 关前叫阵 | 选择提前迎战名将换取奖励，或按正常波次稳守 |
| 战合肥 | 两路疑兵 | 根据军情在两条既有出生路线之间分配防御 |
| 夷陵之战 | 连营火势 | 保护、主动点燃或让敌军引燃营帐，权衡范围伤害和失控风险 |
| 攻克许昌 | 霹雳车 | 定期出现攻城器械，决定优先集火还是继续处理兵线 |
| 攻克建业 | 江潮 | 潮汐周期改变沸水、寒冰和移动减速效果 |

### 7.2 内容制作流程

每张地图单独作为一个 2 万～4 万 Token 子任务：

1. 标定地图坐标和已有路线；
2. 写事件规格和失败边界；
3. 只使用已验收通用动作实现；
4. 添加配置校验和确定性测试；
5. 完成鼠标、触屏、1x/3x 实玩；
6. 记录与军议、将技、Boss 的高风险组合；
7. 更新本表的最终规则和数值。

### 7.3 验收标准

- [ ] 其余六图各有一个独立事件主题。
- [ ] 九图事件不是同一数值模板换名字。
- [ ] 每图事件失败不会导致无法继续普通通关，除非关卡明确标为特殊挑战。
- [ ] 事件奖励在正常经济中不超过预定上限。
- [ ] 所有坐标和路线由自动校验覆盖。

## 8. 里程碑 P3.3：无尽守城

> 状态：待开发
> 建议预算：17 万 Token
> 合理区间：12 万～22 万 Token

### 8.1 解锁和入口

- 完成任一阵营王者之路后开放。
- 标题或自由模式界面增加“无尽守城”入口，不能破坏原版 7 项标题转盘；具体入口在第二阶段结束后评审。
- 玩家选择阵营、武文路线、政令和已解锁地图。
- 每局生成 seed，可复制显示用于复现。

### 8.2 波次结构

- 每 5 波为一个军势周期，第 5 波为名将波。
- 每 10 波提升一个难度阶层，并提供一次无尽军议。
- 敌人数始终受原版 80 槽限制，单波生成上限不超过 79。
- 超过数量上限后主要通过生命、防御、兵种组合、路线和特性提高难度。
- 速度、控制抗性和防御均设硬上限，避免最终变成完全无法反制。
- 名将从已解锁内容池确定性抽取，短周期内不重复。

### 8.3 难度配置

具体公式在开发时以数据表冻结，至少包含：

- 基础生命倍率；
- 防御增长；
- 波次数量增长及上限；
- 名将特性数量或强化层数；
- 多路线概率；
- 波次准备窗口；
- 军议抽取频率和重复层数。

公式必须使用整数或明确舍入，并在第 1、10、25、50、100、200 波建立快照测试。

### 8.4 存档和结束

- 只在安全波间自动保存完整无尽状态。
- 战斗中退出时可恢复到最近安全波间，不保存半波敌人。
- 保存 seed、波数、难度阶层、军议、政令、技能、事件、金币、城防、塔和统计。
- 游戏失败后清除进行中存档，保留最终记录。
- 使用军令可以继续获得本地成绩，但标记为辅助记录，不进入纯净最佳。

### 8.5 验收标准

- [ ] 同 seed 和输入在关键波生成相同内容。
- [ ] 200 波模拟不超过敌人池、不出现数值溢出。
- [ ] 波间恢复和连续运行的第 N 波状态一致。
- [ ] 每波结束后不残留事件订阅、敌人或临时特效。
- [ ] 1～3 倍速不会改变逻辑结果和得分。

## 9. 里程碑 P3.4：每日挑战与挑战规则

> 状态：待开发
> 建议预算：14 万 Token
> 合理区间：10 万～17 万 Token

### 9.1 每日种子

本地每日挑战使用以下稳定输入计算 seed：

~~~text
Asia/Shanghai 日期 YYYY-MM-DD + 内容版本 + 每日规则版本
~~~

相同日期和版本必须生成相同地图、阵营、政令候选、限制、军议和敌军内容。若未来面向其他时区，再把挑战时区改为显式产品配置。

由于第一版没有服务器：

- 不宣称防修改系统时间；
- 允许重复尝试；
- 保存当日最好成绩和尝试次数；
- 可输入历史 seed 进行练习，但历史练习不覆盖当日正式记录。

### 9.2 挑战条件池

第一版从经过可解性审核的条件中抽取 2～3 项：

- 限定可建建筑类型；
- 禁止七级觉醒；
- 初始金币变化；
- 城防上限变化；
- 名将数量增加；
- 普通兵获得轻量词缀；
- 军议池受限；
- 只能选择指定阵营或政令；
- 事件目标升级为强制目标；
- 快速出兵窗口缩短。

禁止随机组合明显互斥或不可解条件，例如“禁止觉醒”同时要求“释放三次将技”。

### 9.3 可解性校验

- 每个条件声明 tags、requires、excludes 和 difficultyWeight。
- 生成器先满足依赖和互斥，再控制总难度权重。
- 每个已发布内容版本保留固定黄金种子集合。
- 自动测试至少验证连续 365 个日期配置合法且无空内容池。
- 不能用 AI 主观判断代替配置约束和自动校验。

### 9.4 验收标准

- [ ] 相同日期、版本在刷新和不同设备尺寸下内容一致。
- [ ] 连续 365 日生成测试无非法组合。
- [ ] 最佳成绩按日期和内容版本保存。
- [ ] 练习 seed 与正式每日记录严格区分。
- [ ] 经典规则和原版存档不受每日挑战影响。

## 10. 里程碑 P3.5：计分、记录、长稳测试与总验收

> 状态：待开发
> 建议预算：15 万 Token
> 合理区间：14 万～23 万 Token

### 10.1 计分原则

计分只依赖战斗逻辑和玩家结果，不依赖显示刷新率、设备性能或真实渲染帧数。建议组成：

- 完成波次基础分；
- 普通敌人和名将击杀分；
- 剩余城防奖励；
- 连战和快速出兵奖励；
- 历史事件目标奖励；
- 挑战难度倍率；
- 未使用军令的纯净标记；
- 可选的金币效率或塔投资效率奖励。

不建议直接按现实耗时奖励，否则暂停、后台和设备差异难以公平。若需要时间因素，使用“波次准备逻辑时间”或其他可控指标。

### 10.2 分数版本

- ScoreVersion 独立于内容版本。
- 修改公式后旧记录保留旧版本标签，不与新记录直接比较。
- 分数计算使用整数，固定运算顺序。
- 结算展示分项来源，不能只显示一个总数。
- 辅助记录与纯净记录分栏。

### 10.3 长稳与性能测试

至少执行：

- 无渲染 200 波模拟，多组固定 seed；
- 3 倍速高敌人数实机运行；
- 每波保存/加载 50 次往返；
- 九图事件分别连续重开和继续游戏；
- 每日挑战 365 日期生成；
- 军议、将技、Boss、事件同时活跃的压力场景；
- 浏览器 localStorage 容量与损坏数据恢复测试。

### 10.4 总体验收

- [ ] P3.0～P3.4 全部验收完成。
- [ ] 九图各有独立事件并可在强化王者之路通关。
- [ ] 无尽模式稳定模拟 200 波。
- [ ] 每日挑战相同日期和版本可复现。
- [ ] 分数分项、版本和辅助标记正确。
- [ ] npm test 与 npm run build 通过。
- [ ] 经典模式完整战役流程无新增入口阻塞或数值变化。
- [ ] 桌面和移动端完成无尽、每日、事件各一轮操作验收。
- [ ] 更新总纲最终状态、实际 Token、已知限制和后续候选。

| 移动目标影响原版寻路 | 独立事件实体和预设节点 | 改成静态保护目标 |
| 无尽数值溢出 | 整数上限、波次快照、200 波测试 | 进入封顶循环 |
| 每日组合不可解 | tags/依赖/互斥/黄金种子 | 使用当天保底固定方案 |
| localStorage 过大 | 精简状态、记录数量上限 | 只保留最佳和最近记录 |
| 内容更新破坏复现 | 内容/随机/分数独立版本 | 旧记录只读保留 |
